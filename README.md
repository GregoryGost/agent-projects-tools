# agent-projects-tools

`agent-projects-tools` — единое хранилище переносимых правил и скилов для AI-агентов, используемых в разных проектах.

## Назначение

Репозиторий нужен, чтобы хранить в одном месте проектно-независимые материалы для AI-агентов:

1. правила поведения и работы агента;
2. скилы, которые можно переиспользовать между проектами;
3. инструкции, соглашения и шаблоны, не привязанные к конкретному кодовому стеку.

Материалы из этого репозитория должны быть переносимыми: их можно копировать, подключать или адаптировать в других проектах без переноса исходного кода приложений.

## Текущий фокус

Сейчас репозиторий ориентирован на материалы для Codex. Поддержка других AI-агентов может быть добавлена позже отдельными директориями или соглашениями, если в этом появится необходимость.

## Что не входит в проект

Этот репозиторий не предназначен для хранения кода приложений. В нём также не предполагаются команды сборки, запуска или тестирования приложения, потому что проект выполняет роль библиотеки правил и скилов, а не исполняемого программного продукта.

## Структура репозитория

Текущая структура минимальна:

```text
.
├── .gitkeep
└── README.md
```

По мере добавления материалов структура может быть расширена. Например, материалы можно группировать по агентам, типам правил или скилам:

```text
.
├── codex/
│   ├── rules/
│   └── skills/
└── README.md
```

Итоговая структура должна отражать фактическое содержимое репозитория и обновляться вместе с ним.

## Использование

Базовый сценарий использования:

1. выберите нужное правило, скил или инструкцию;
2. перенесите материал в целевой проект или подключите его способом, который поддерживает используемый агент;
3. при необходимости адаптируйте только проектно-специфичные части;
4. сохраняйте общие и переиспользуемые изменения обратно в этом репозитории.

## Принципы ведения материалов

При добавлении новых правил и скилов рекомендуется соблюдать следующие принципы:

1. **Переносимость** — материал не должен зависеть от внутренней структуры одного конкретного проекта, если это не указано явно.
2. **Ясная область применения** — должно быть понятно, для какого агента и сценария предназначен материал.
3. **Минимум неявных предположений** — требования, ограничения и зависимости нужно описывать явно.
4. **Актуальность** — устаревшие правила и скилы следует обновлять или удалять.

## Участие в разработке

Для изменения репозитория:

1. создайте ветку от актуальной `master`;
2. добавьте или обновите правила, скилы либо документацию;
3. проверьте, что изменения не привязаны к одному проекту без необходимости;
4. откройте pull request с кратким описанием назначения изменений.
